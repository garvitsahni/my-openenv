
WORKBENCH
AI Agent Training Environment Platform
Product Requirements Document  ·  Hackathon Edition



1. Executive Summary
VISION  WorkBench is the missing infrastructure layer for AI agent research — a community-curated, OpenEnv-compliant library of real-world training environments, each deployable in one click to Hugging Face Spaces.

The AI agent research community has a fundamental infrastructure problem. Every team building LLM agents — whether at universities, startups, or enterprise labs — must first spend weeks constructing evaluation environments from scratch. These environments are incompatible, unreproducible, and nearly always simulate toy problems that bear no resemblance to real work.
WorkBench solves this by providing three things in one submission: (1) a production-quality real-world training environment, (2) a fully compliant OpenEnv server, and (3) a reliable baseline inference script that runs deterministically on constrained hardware.


2. Problem Statement
2.1  The Core Problem
Imagine you are a machine learning researcher who wants to train an LLM agent to handle a real-world task — triaging an inbox, reviewing code, or cleaning a messy CSV file. Before you can run a single training step, you must build the environment yourself: the state machine, the action space, the observation format, the reward function, the deployment scaffolding, and the evaluation harness. This typically takes two to three weeks.
When your paper is published, nobody else can reproduce your results because they cannot recreate your environment. When another team builds a better agent, you cannot compare it to yours because they used a different environment. The field makes progress, but it cannot measure that progress reliably.

6. System Architecture
6.1  Architecture Overview (4 Layers)
The system is organized into four layers. Each layer has a single responsibility and depends only on the layer below it.


6.2  Data Flow
1. Task seed JSON is loaded into the environment on POST /reset. Episode ID is generated and returned with the initial observation.
2. The inference script reads observation + history, builds a structured prompt, calls the LLM, parses and validates the response, and sends the action to POST /step.
3. POST /step dispatches the action to the environment state machine, updates episode state, runs the grader, and returns (observation, reward, done, info).
4. All step data is appended to a JSONL session log. The episode continues until done=True or max_steps is reached.
5. GET /score/{episode_id} returns the final score, per-step breakdown, and grader details for the complete episode.

10. Scalability & Future Scope
10.1  Scaling Path
Phase 1 (Hackathon): Single HF Space container per environment. All state in-process. Handles ~10 concurrent episodes.
Phase 2 (Post-launch): Extract environment runtime and grader engine into stateless function-as-a-service (HF Inference Endpoints or AWS Lambda). Episode state migrates to Redis. This enables horizontal scaling to thousands of concurrent episodes without architecture rewrite — the API surface stays identical.
Phase 3 (Scale): Multi-region deployment. CDN-cached task fixtures. Dedicated grader workers for compute-intensive tasks. Async episode execution for batch evaluation workflows.

2. env/models.py — PYDANTIC MODELS
   Define strictly typed models:
   - EmailAction(BaseModel): action_type: Literal[
       'classify','assign','escalate','draft','skip','done']
     args: dict with validated fields per action_type
     episode_id: str
   - EmailObservation(BaseModel): emails: list[Email]
     step_count: int, remaining_steps: int,
     available_actions: list[str], context: dict
   - Email(BaseModel): id, subject, sender, body, timestamp,
     thread_id, attachments: list
   - GraderResult(BaseModel): score: float (0.0-1.0),
     breakdown: dict, criteria_met: list[str],
     max_possible: float

3. env/environment.py — EMAIL TRIAGE ENVIRONMENT
   class EmailTriageEnv:
     def reset(task_id: str, seed: int = 42) -> EmailObservation:
       Load task JSON, initialize episode state, return obs
     def step(action: EmailAction) -> tuple[EmailObservation,
               float, bool, dict]:
       Validate action, update state, call grader,
       return (obs, reward, done, info)
     def state() -> dict:
       Return full current state including episode metadata
   Rules:
   - Episode ends when agent calls 'done' or max_steps=30 reached
   - Each action produces an immediate partial reward
   - Observation always shows remaining emails and step count
   - All state must be deterministic given same seed

4. env/grader.py — DETERMINISTIC GRADER
   class EmailTriageGrader:
     def score_action(action, email, ground_truth) -> GraderResult:
       Evaluate single action against rubric. Return partial credit.
     def score_episode(episode_log) -> float:
       Aggregate all step scores. Return final 0.0-1.0 score.
   Grading rubric:
   - Correct classification: +0.15 per email
   - Correct team assignment: +0.10 per email
   - Correct escalation flag: +0.10 per escalation-required email
   - Response draft quality (easy+medium only): +0.05 per email
   - Correctly identifying phishing as spam (hard only): +0.25 bonus
   - PENALTY: flagging non-escalation as escalation: -0.05
   - PENALTY: marking phishing as urgent: -0.15
   All penalties and bonuses must be deterministic and rubric-based.

5. tasks/*.json — TASK SEEDS
   Generate realistic email fixtures for each difficulty level.
   Each seed contains:
   - 10 emails with realistic subjects, senders, body text
   - ground_truth: { email_id: { category, team, escalate,
                    response_hint } }
   - metadata: { difficulty, seed, max_steps, passing_score }
   Hard tier must include:
   - 1 phishing email disguised as urgent IT request
   - 2 emails with conflicting priority signals
   - 1 duplicate thread (same issue, two senders)

6. main.py — FASTAPI SERVER
   Implement all 5 OpenEnv endpoints:
   POST /reset  → { observation, episode_id, task }
   POST /step   → { observation, reward, done, info }
   GET  /state  → { state, step_count, cumulative_score }
   GET  /tasks  → list of available tasks
   GET  /score/{episode_id} → final score + breakdown
   Also implement: GET /health (returns 200 for HF ping check)
   Use dependency injection for environment instance.
   Handle all validation errors with proper HTTP status codes.

7. openenv.yaml
   name: workbench-email-triage
   version: 1.0.0
   description: Real-world email triage environment for LLM agents
   action_space: { type: discrete, actions: [classify, assign,
                   escalate, draft, skip, done] }
   observation_space: { type: structured, fields: [emails,
                        step_count, available_actions] }
   tasks: [easy, medium, hard]
   reward_range: [-0.15, 0.25]  # min step reward, max step reward
   max_steps: 30
   tags: [email, triage, real-world, nlp]

8. inference.py — BASELINE INFERENCE SCRIPT
   Requirements:
   - Must use OpenAI client: openai.OpenAI(base_url=..., api_key=...)
   - Reads from env vars: API_BASE_URL, MODEL_NAME, HF_TOKEN
   - Runs a full episode against all 3 difficulty tiers
   - Prints per-step reward and final score for each tier
   System prompt MUST enforce strict JSON output:
     "You are an email triage agent. Respond with ONLY a single
      JSON object matching this schema exactly, no other text:",
     { "action_type": "<classify|assign|escalate|draft|skip|done>",
       "args": { ... } }
   Implement these 4 reliability features:
   a) parse_action(raw): strips markdown fences, extracts JSON,
      returns None on failure (do NOT raise)
   b) validate_action(parsed): checks action_type in allowed list,
      checks required args present for that action_type
   c) call_with_retry(prompt, max_retries=3): on None/invalid,
      appends error hint to prompt and retries. After 3 failures,
      returns safe fallback: {'action_type': 'skip', 'args': {}}
   d) build_prompt(obs, history): formats observation + last 3
      steps as structured context. Ends with exact JSON schema.
   Log every step to JSONL: step_number, action, reward,
   cumulative_score, raw_response_preview (first 100 chars).
   The script must complete ALL 3 tiers in under 20 minutes on
   a 2 vCPU / 8 GB machine with no GPU.
